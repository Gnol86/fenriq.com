# Custom ESLint Rules for Boilerplate Protection

## Overview

This directory contains custom ESLint rules that protect boilerplate files from accidental modification in derived projects.

## Rule: `no-edit-boilerplate`

### Purpose

Prevents developers from editing boilerplate-managed files when working in a derived project (Project A, B, C, etc.).

### How It Works

The rule automatically detects:

1. **Is this a project?** - Checks if the repository has an `upstream` git remote
   - If YES → Rule is active (protection enabled)
   - If NO → Rule is inactive (you're in the boilerplate itself)

2. **Is this a boilerplate file?** - Checks if the file matches protected patterns:
   - `src/actions/*.action.js`
   - `src/components/**` (except `src/components/project/`)
   - `src/lib/**`
   - `src/hooks/**`
   - `src/app/(pages)/**`
   - `src/app/(auth)/**`
   - `src/app/layout.js`
   - `prisma/schema/base.prisma`
   - `src/messages/{fr,en,nl,de}.json`

3. **If both conditions are true** → ESLint error is thrown

### Example Error

When you try to edit a boilerplate file in a project:

```
src/components/ui/button.jsx
  1:1  error  🚨 BOILERPLATE FILE - Do not edit this file in your project!
              This file is managed by the boilerplate and will be overwritten on updates.
              Put your project code in: src/project/
              See: .github/SETUP_NEW_PROJECT.md
              boilerplate/no-edit-boilerplate
```

### Protected Files

**✅ Safe to edit** (never triggers error):
- `src/project/**` - All project code
- `src/messages/*.project.json` - Project translations
- `prisma/schema/project.prisma` - Project database models
- `src/app/page.js` - Landing page template
- `src/app/globals.css` - Theme/styling
- `src/site-config.js` - Site configuration
- Any files outside the patterns above

**🚨 Protected** (triggers error in projects):
- All boilerplate source code
- Boilerplate translations
- Base Prisma schema
- Dashboard and auth pages

### Disabling the Rule

If you absolutely need to disable this rule (not recommended):

**For a specific file:**
```js
/* eslint-disable boilerplate/no-edit-boilerplate */
// Your code here
```

**For a specific line:**
```js
// eslint-disable-next-line boilerplate/no-edit-boilerplate
const myVar = "something";
```

**Globally (in eslint.config.mjs):**
```js
rules: {
  "boilerplate/no-edit-boilerplate": "off", // or "warn"
}
```

### Testing the Rule

**In the boilerplate repository:**
```bash
# No errors - rule is inactive (no upstream remote)
pnpm lint
```

**In a derived project:**
```bash
# Setup project with upstream remote
git remote rename origin upstream
git remote add origin https://github.com/you/your-project.git

# Edit a boilerplate file (will fail)
echo "// test" >> src/components/ui/button.jsx
pnpm lint
# Error: 🚨 BOILERPLATE FILE - Do not edit...

# Edit a project file (works fine)
echo "// test" >> src/project/components/my-component.jsx
pnpm lint
# No errors
```

### Implementation Details

**File:** `eslint-rules/no-edit-boilerplate.js`

The rule:
1. Runs a shell command `git remote` to check for `upstream`
2. Compares the current file path against regex patterns
3. Reports an ESLint error if both conditions match

**Performance:** Minimal impact - git command is cached per ESLint run.

### Adding New Protected Patterns

Edit `BOILERPLATE_PATTERNS` in `no-edit-boilerplate.js`:

```js
const BOILERPLATE_PATTERNS = [
    /^src\/actions\/.*\.action\.js$/,
    /^src\/components\/(?!project\/).*$/,
    // Add your pattern here
    /^src\/new-folder\/.*$/,
];
```

**Pattern Tips:**
- Use regex for flexible matching
- Use negative lookahead `(?!project\/)` to exclude project folders
- Test patterns thoroughly

## Advantages

✅ **Prevents mistakes** - Catch boilerplate edits before committing  
✅ **IDE integration** - Real-time feedback in VS Code, etc.  
✅ **No git hooks needed** - Works directly in your editor  
✅ **Self-documenting** - Error message explains what to do  
✅ **Zero config** - Automatic detection of project vs boilerplate  

## Limitations

⚠️ **Requires git** - Won't work if not using git  
⚠️ **Remote name dependency** - Assumes `upstream` remote name  
⚠️ **Can be disabled** - Developers can bypass with eslint-disable comments  

## Alternatives

If this approach doesn't work for you:

1. **Git hooks** - Use pre-commit hooks to check modified files
2. **CI/CD checks** - Verify in GitHub Actions/GitLab CI
3. **Documentation only** - Rely on developer discipline
4. **Separate repos** - Keep boilerplate and projects completely separate
